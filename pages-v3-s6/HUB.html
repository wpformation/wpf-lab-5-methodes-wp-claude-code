// LE PROBLÈME · 2026

WordPress n'a jamais offert autant de chemins pour construire une page. Gutenberg natif, Gutenberg complété de CSS personnalisé, HTML brut monobloc ou multi-blocs, plugin UI Spectra Pro avec ses containers et ses presets. En 2026, on a le choix. Trop, peut-être.

Chaque méthode promet d'être la meilleure. Aucune ne l'est dans l'absolu. La bonne question n'est pas « quelle est la meilleure ? » mais « la meilleure pour quoi, et au prix de quoi ? ».

Pour trancher, on a fait quelque chose de simple : on a construit la même page sept fois, avec les sept méthodes, et on a scoré chaque résultat sur quatre critères concrets qui comptent vraiment quand on doit livrer un site à un client (ou maintenir le sien sur cinq ans). Les démos sont publiées. Vous pouvez les voir, les cliquer, ouvrir leur code source. Et juger.

Sur quoi on les a notées.

P·01

Éditabilité native

P·02

Qualité design

Le rendu visuel est-il éditorial, sur-mesure, brutalist — ou wireframe générique de builder par défaut ?

P·03

Indépendance plugin

La page survit-elle si on désactive le plugin builder ? Ou tout casse en visuel et en markup ?

P·04

Reproductibilité

Un développeur peut-il rejouer la même méthode sur 10 sites différents ? Ou est-ce magique-unique-jamais-reproductible ?

M · 01

accueil-gutenberg-natif

Gutenberg natif

Builder par défaut · 0 plugin tiers · 100 % core

La voie la plus orthodoxe : on utilise uniquement les blocs core/* livrés avec WordPress. Chaque paragraphe, chaque titre, chaque colonne reste éditable un à un dans l'éditeur. Maximum éditable, minimum visuel.

  • Aucune dépendance externe
  • Éditable par n'importe quel rédacteur
  • Design contraint par les blocs core

4.0/5

voir la démo ↗︎

M · 02

methode-wpformation

Gutenberg + CSS personnalisé

Voie d'Alexandra · CSS injecté page-par-page

Blocs natifs Gutenberg + CSS sur-mesure injecté via meta page-level (Spectra). Chaque paragraphe reste modifiable. Le design est totalement libre, brutalist, sur-mesure. Le compromis parfait éditabilité × design.

  • Éditabilité Gutenberg complète préservée
  • Design 100 % sur-mesure CSS
  • Reproductible par n'importe quel dev front

5.0/5

voir la démo ↗︎

M · 03

methode-html-monobloc

HTML monobloc

1 seul bloc · full design control · 0 éditabilité

Tout le markup HTML dans un unique bloc wp:html. Design 100 % maîtrisé, éditabilité quasi-nulle pour un rédacteur. Idéal pour landing one-shot ou intégration imagée livrée par un graphiste. Design libre, éditabilité sacrifiée.

  • Rendu pixel-perfect garanti
  • Aucune contrainte de bloc
  • Inéditable depuis l'admin WP

3.0/5

voir la démo ↗︎

M · 04

methode-html-multiblocs

HTML multi-blocs

10 blocs séparés · sections modulables · HTML opaque

Chaque section dans un wp:html distinct. Compromis entre M·03 et M·02 : on peut réorganiser ou retirer des sections depuis l'éditeur, mais le code de chaque section reste opaque pour un rédacteur. Section-par-section, mais toujours du HTML.

  • Sections déplaçables dans l'éditeur
  • Suppression sélective possible
  • Code HTML toujours opaque

3.5/5

voir la démo ↗︎

M · 05

methode-spectra-pro

Spectra Pro

Plugin UI · containers + InfoBox + Buttons

On utilise les blocs uagb/* du plugin Spectra Pro : containers responsifs, InfoBox prêts-à-l'emploi, Buttons avec options avancées. UI riche dans l'éditeur, mais dépendance forte au plugin payant. UI riche, dépendance plugin.

  • Options layout responsives natives
  • Presets visuels intégrés
  • Dépendance au plugin Spectra Pro

3.5/5

voir la démo ↗︎

M · 06

methode-gutenberg-pur

Gutenberg pur + CSS dans core/html

100 % core · 1 seul bloc core/html style · 0 plugin

Variante 100 % native de la voie d'Alexandra : blocs core/* exclusivement, le CSS sur-mesure est embarqué dans un seul bloc core/html en tête de page, pas en meta Spectra. Édition Gutenberg native sur tout le contenu textuel. Indépendance maximale, content auto-suffisant.

  • Aucun plugin tiers
  • CSS visible et éditable depuis l'admin
  • Content portable (export = page complète)

5.0/5

voir la démo ↗︎

 ★ LA NOUVELLE · BLOCS PHP NATIFS WP 7.0

M · 07

methode-blocs-php-custom

Blocs PHP custom (autoGenerateControl WP 7.0)

Plugin wpf-lab · 11 blocs PHP · 0 JS · 0 build · 0 ACF

Plugin custom wpf-lab embarquant 11 blocs Gutenberg en pur PHP. Chaque attribut autoGenerateControl: true du block.json génère automatiquement la commande UI dans le panel WP — sans une ligne de JavaScript ni de build. Design 100 % locké, édition non-destructive. La voie pro pour design system + rédacteurs non-techs.

  • Design 100 % verrouillé dans le plugin
  • Édition non-destructive (champs typés)
  • Versionnable Git, headless-ready

5.0/5
